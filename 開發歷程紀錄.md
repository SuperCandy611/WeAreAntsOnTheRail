## 一、為什麼選這個題目

> 📝 *以下為草稿，請根據你的實際狀況修改*

總之，想找可以反映臺灣人日常生活的資料。

我向家姊尋求意見，他邊滑手機邊說了句「螞蟻！」

於是，想做交通人流的概念就這樣誕生了。如果從空中觀察，我們的通勤、遷徙人流，大概就像個巨大的蟻箱吧。

每逢春節、端午、中秋，臺灣各地的火車站總是人聲鼎沸。有些人從台北出發回老家，有些人從南部北上探親，這種「大遷徙」的現象讓我很好奇：**如果把台鐵每天的進出站資料畫出來，能不能看到這股人潮到底是怎麼流動的？**

剛好台鐵有公開的每日進出站人次資料，而且時間跨度長達二十年（2005–2026），還橫跨了 COVID-19 疫情這個重大事件，資料本身就有很多值得探索的角度。

於是決定以此作為期末作業，用 Python + Streamlit 做一個互動式的分析儀表板。

## 二、資料收集

### 主要資料來源

所有資料均來自**台灣鐵路管理局**的公開資料集，透過政府開放資料平台（data.gov.tw）下載。

| 資料集 | 格式 | 時間範圍 | 筆數 |
|--------|------|----------|------|
| 每日各站進出站人數（舊格式） | CSV × 3 | 2005–2019/04/22 | 約 113 萬筆 |
| 每日各站進出站人數（新格式） | CSV × 7 | 2019/04/23–2025 | 若干萬筆 |
| 每日各站進出站人數（2026） | JSON | 2026/01–2026/05 | 約 2.9 萬筆 |
| 車站基本資料集 | JSON | 現況 | 245 站 |

另外，三節連假的日期並非直接在台鐵資料裡，我自己整理了一份：

- **`taiwan_holidays_and_summer_vacations_2005_2026.csv`**：手動查詢並整理 2005–2026 年的春節、端午、中秋連假日期區間，以及暑假起訖日。

- **`supp_info.csv`**：台鐵的車站 JSON 裡只有站名和 GPS，沒有「縣市」、「對號/非對號」這些分類資訊。這份檔案是我自己一站一站對照查找、手工整理的，共涵蓋 240 個車站，記錄了每站所在縣市與是否為對號列車停靠站。

## 三、資料結構：踩到第一個坑

拿到資料之後，第一件事是搞清楚格式。台鐵的資料在 2019 年 4 月有一次格式大改版，這讓整合工作複雜了不少：

### 舊格式（2005–2019/04/22）

```
BOARD_DATE, TKT_BEG, STOP_NAME, 進站, 出站
20050101, 2, 基隆, 3421, 2980
```

- 站碼（`TKT_BEG`）是**內部流水號**（2、4、6 … 292），跟其他資料對不起來
- 好消息是同一行有**站名**（`STOP_NAME`），可以當橋梁

### 新格式（2019/04/23–2026）

```
trnOpDate, staCode, gateInComingCnt, gateOutGoingCnt
20190423, 900, 5100, 4890
```

- 站碼（`staCode`）直接對應車站 JSON，整合很方便
- 2026 年的資料改成 JSON 格式，但欄位內容相同

### 解決方式

舊格式有站名，新格式/JSON 有站碼。因此用**站名**作為共同鍵：

1. 從車站 JSON 建立「站名 → staCode」的對照表
2. 舊格式透過站名查到 staCode，統一轉換
3. 最後全部合併成一張大表，欄位統一為 `date / staCode / in_count / out_count`

## 四、資料前處理

整合完成後，還需要把「站碼」和「車站元資料」（GPS、縣市、車種）接回來。

### 合併流程

```
原始資料（3種格式）
    ↓ 統一欄位名稱
    ↓ 日期轉 Timestamp
    ↓ 以站名對照 staCode（舊格式）
df_all（staCode + date + in/out）
    ↓ merge 車站基本資料集.json（GPS座標）
    ↓ merge supp_info.csv（縣市、對號/非對號、主線/支線）
最終 df（約 120 萬筆，含所有欄位）
```

### 衍生欄位

- **`net_flow`**（淨流量）= `in_count - out_count`
  - 正值：進多出少 → 此站為出發地（人正在離開）
  - 負值：出多進少 → 此站為目的地（人正在抵達）

### 效能考量：`@st.cache_data`

資料量約 120 萬筆，每次重新整合要跑 10–20 秒。Streamlit 的 `@st.cache_data` 裝飾器讓資料只在首次載入時計算一次，之後切換頁面都是秒開。

## 五、踩過的坑：臺 vs 台 的字形戰爭

這是整個開發過程中最意外的一個 bug，值得特別記錄。

### 問題描述

頁面 2、3、4 的統計圖表，原本應該出現的台北、台中、台南、台東等大站，全部消失了，但頁面 1 的地圖卻能正常顯示。

### 根本原因

台鐵的車站 JSON 裡用的是「**臺**」字（臺北、臺中、臺南、臺東），而我自己整理的 `supp_info.csv` 一開始用的是「**台**」字。

`data_loader.py` 在讀車站 JSON 時，有一個 `_NAME_NORM` 對照表把「臺」統一轉成「台」，所以 `df_sta` 裡的站名都是台形。但後來我為了讓縣市名稱顯示正確（台北市 → 臺北市），把 `supp_info.csv` 裡的站名改成了臺形。

結果：
- `df_sta` 的站名：`台北`
- `supp_info` 的站名：`臺北`
- `merge` 對不起來 → `city / tkt_type / line_type` 全部變 `NaN`
- `groupby` 預設會丟掉 NaN 的列 → 大站消失

### 解決方式

在 `data_loader.py` 的 merge 前，對 `supp_info` 的站名也套用 `_NAME_NORM`，讓兩邊統一成台形，merge 才能成功。

```python
supp["stationName"] = supp["stationName"].replace(_NAME_NORM)
```

## 六、功能設計：四個分析角度

規劃了四個分頁，每個對應一個問題：

| 分頁 | 核心問題 |
|------|----------|
| 🗺 節假日流動地圖 | 這個假期，哪些站是出發地？哪些是目的地？ |
| 📈 車站 20 年趨勢 | 這個站的客流量，在 20 年間怎麼變化？ |
| 🏙 區域流量比較 | 不同縣市在同一假期，流量差異有多大？ |
| 🦠 COVID 衝擊觀察 | 疫情把哪些站打得最重？哪些最抗跌？ |

## 七、逐頁開發說明

### 頁面一：節假日流動地圖

這是整個儀表板的主打頁面，也是最花心思的一頁。

**技術：PyDeck ScatterplotLayer**

選用 pydeck 而不是 folium 或 plotly 地圖，是因為 pydeck 的泡泡圖在大量資料點時渲染更流暢，且支援客製化 tooltip。

**顏色邏輯：**
- 紅色 = 淨流出（`net_flow > 0`，進多出少）= 人從這裡出發的城市
- 藍色 = 淨流入（`net_flow < 0`，出多進少）= 人抵達的目的地
- 泡泡大小 = 進出站總人次（絕對流量）

一開始搞反了紅藍的定義，後來想清楚：春節前台北人往南走，台北站是「進站人多」（大家提著行李進站搭車出發），所以台北應該是紅色（淨流出出發地）。

**互動功能：**
- 假期類型 × 年份 × 日期滑桿
- 單日地圖 / 假期累計切換
- 僅對號車站篩選

### 頁面二：車站 20 年趨勢

這頁比較直觀，難點在選站介面的設計。

一開始是一個下拉選單列出所有站，但 245 個站捲到眼睛花。後來改成**兩層選擇**：先選縣市，再選該縣市的車站，大幅提升體驗。

COVID 期間（2020–2022）用半透明橘色矩形標記，讓折線圖的「斷崖」一目了然。

### 頁面三：區域流量比較

**比較年份圖例問題（已修正）：**

最初年份是用整數當 `color` 欄位，Plotly 把它當連續數值，跑出來的圖例是一條連續色帶，看不清楚哪根bar是哪一年。修正方式是把年份轉成字串，Plotly 就會改用分項圖例（黃色 = 2021 這種）。

```python
region_df["year"] = region_df["year"].astype(str)
```

**分析時段選項：**

原本「進站人次」和「出站人次」畫出來幾乎一樣，後來加了「假期首日 / 假期末日」選項，才能看到明顯差異——首日台北是出發地，末日台北是目的地。

**為何最多選 5 年：**

比較年份使用 grouped bar chart，X 軸是各縣市，每個縣市下有多根 bar 並排。年份越多，每根 bar 越細，超過 5 個基本上看不清楚，所以限制 5 年是視覺實用上限。

### 頁面四：COVID 衝擊觀察

**Bar Chart Race 動畫：**

這是最有趣的功能，用 Plotly 的 `go.Figure(frames=...)` 實作，讓各站的年度運量排名「動起來」。固定顯示歷年累計排名前 15 的車站，播放時能看到名次的升降變化。

**跌幅分析（已新增對號車站篩選）：**

比較各站 2019 vs 2020 年的進站量，計算跌幅百分比。加入「僅對號車站」勾選後，可以過濾掉小站的雜訊，專注觀察城市大站的衝擊差異。

## 九、心得與回顧

> 📝

這次作業花最多時間的，意外地不是「畫圖」，而是**整理資料**。

光是讓三種格式的資料能夠合併、讓站名字形一致、讓縣市資訊能夠正確接回去，就花了將近一半的開發時間。但這也讓我更深刻地理解：資料科學裡有句名言「80% 的時間在清資料」，原來不是誇張，是真的。

另一個收穫是學會了 `@st.cache_data` 的重要性。第一版的 app 沒有快取，每次切換頁面都要重新跑一遍所有計算，慢到讓人崩潰。加上快取之後，整個體驗天差地別。

最後讓我驚喜的是 Bar Chart Race 動畫，用 Plotly 的 frames 機制不到 100 行就做出來了，看著台北站二十年穩坐第一名的樣子，還有 COVID 期間所有站一起崩跌，有種說不出的震撼感。

至於看到成果的心得嘛，只有說不出的感動，看到2025年中秋超過歷年春節的瘋狂流量，就想起自己當初搶不到票只能一路站著搖回老家的心情，覺得這筆資料真的和自己身處的時空連結再一起了，對我來說，這就是最有意義的事。
